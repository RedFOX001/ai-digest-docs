Обзор

Это n8n-воркфлоу, который каждую ночь забирает историю Telegram-чата поддержки за прошедшие МСК-сутки, прогоняет через 5 LLM-вызовов с двумя независимыми критиками и верификацией по официальной документации, и публикует готовый структурированный дайджест — в чат модераторов и в публичный канал.

Ключевое отличие от простых AI-суммаризаторов — проверяемость: каждая тема обязана иметь ссылку на реальное сообщение, каждое решение проверяется отдельным LLM-критиком, а затем сверяется с документацией проекта.

Как это работает

  1. Сбор и фильтрация данных
    Workflow забирает историю чата через Telegram GramPro API, отфильтровывая сообщения ботов, предыдущие дайджесты (по маркеру 📋 Дайджест и 🔗) и сообщения вне временного окна. После фильтрации формируется структурированный лог с индексом сообщений и списком anchor-кандидатов — сообщений с признаками жалоб.
  2. Извлечение тем (LLM #1 — DeepSeek V4 Flash)
    Модель получает индекс, anchor-кандидаты и полный лог чата. Извлекает до 12 проблем, 5 советов и релизы по строгим Tier-правилам. Поле solution на этом этапе запрещено — проставляется только при явном подтверждении от автора темы.
  3. Критик #1 (LLM #2)
    Второй LLM-вызов проверяет JSON от экстрактора: удаляет слабые темы, сливает дубли по первопричине, валидирует id по индексу сообщений, проверяет дубликаты. Если критик вернул менее 50% тем — срабатывает fallback на данные до критика.
  4. Критик #2 per-topic (LLM #3)
    Отдельный LLM-запрос на каждую тему с контекстным окном (20 сообщений до anchor, 80 после + все сообщения автора темы). Ищет кросс-юзер конфликты, проверяет соответствие решения логу, валидирует заголовок. Принудительно восстанавливает оригинальный id, если LLM его изменил.
  5. Верификация по документации (LLM #4)
    Четвёртый LLM-вызов сверяет каждое решение/совет с официальной документацией проекта (README + FAQ с GitHub). verified=true — только если в документации есть точное совпадение. При конфликте (маркер «расходятся») — принудительно verified=false.
  6. Финальный генератор (LLM #5)
    Пятый LLM-вызов собирает HTML-текст для Telegram по жёсткому шаблону с разделами «Главное за день», «Новости и релизы», «Проблемы и решения», «Советы дня». Каждый пункт содержит ссылку на исходное сообщение.
  7. Код-валидатор (0 токенов, без LLM)
    Детерминированная проверка финального текста: замена Markdown на HTML, проверка баланса тегов, удаление пустых разделов, проверка на заглушки, схлопывание лишних пустых строк.
  8. Логгирование и отправка
    В таблицу digest_runs (Supabase/PostgreSQL) записывается полная информация о запуске: статусы этапов, токены, стоимость, ошибки. Затем дайджест отправляется в чат модераторов и, после очистки диагностики, в публичный канал.

🎯 Правила отбора тем: Tier 1 и Tier 2

Система фильтрует сообщения по двухуровневой системе приоритетов до того, как LLM начнёт извлекать темы. Это не решение модели — это жёсткие правила в промпте экстрактора, которые отсекают шум и выделяют то, что действительно стоит внимания модератора.

Tier 1 — ВСЕГДА включать

Даже при одном участнике и без ответов, если есть любой из маркеров:

Маркер Пример НЕ пример
Модель устройства + тех. симптом KN-2310 + SIGSEGV «у меня KN-1810, всё ок»
Код ошибки + контекст curl response code: 400 в логе xkeen «ошибка какая-то»
Версия + явная проблема xray 26.6.1 + вылетает «версия последняя, нормально»

Допустимые симптомы в Tier 1: ошибка, краш, не стартует, SIGSEGV, 504, таймаут, зависает.
НЕ считаются симптомами: «не работает», «проблема», «не получается».

Tier 2 — Включать при наличии обсуждения

Достаточно одного из критериев:

3+ участника

Минимум 3 автора в теме

2+ с конфигом/скриншотом

Детальное техническое обсуждение

1+ с конкретным решением

Модератор или автор предложили решение

Инфраструктурная ситуация

РКН, VPS недоступен, бан IP — даже при 2 участниках

⚠️ Важно: подтверждение от автора может прийти значительно позже в логе (10–100 сообщений спустя). Тема не считается «без подтверждения», если ответ автора просто далеко от жалобы.

Что НЕ включается в дайджест

  • ❌ 1 участник + нет ответов + нет маркеров Tier 1
  • ❌ Общие вопросы без симптома («как настроить X?», «где почитать?»)
  • ❌ Упоминание модели/версии без проблемы («у меня KN-1810, всё ок»)
  • ❌ Tip с одной голой ссылкой без описания ЧТО делает инструмент
  • ❌ Решение/воркэраунд от третьего лица без подтверждения автора темы

Архитектурные особенности

  • Факт-чекинг через id Каждая тема обязана иметь ссылку на реальное сообщение в чате. LLM не может добавить факт без подтверждения — id проверяется по индексу сообщений на нескольких этапах.
  • Детекция кросс-юзер конфликтов Если один пользователь пишет «помогло», а другой в той же теме «не работает» — Critic #2 per-topic понижает статус до «предложено» с маркером «результаты расходятся».
  • Верификация по официальной документации Четвёртый LLM-вызов сверяет каждое решение с README и FAQ проекта. Подтверждённые и неподтверждённые решения явно разделены (verified true/false).
  • Принудительная защита id Нормализатор ответов Critic #2 принудительно восстанавливает оригинальный id, если LLM его изменил — несмотря на прямой запрет в промпте.
  • Fallback-система При сбое любого LLM-этапа используются данные предыдущего этапа. Критик #1 работает с continueOnFail. Дайджест не теряется при единичных сбоях.
  • Прозрачная диагностика В футере каждого дайджеста — статус по каждому этапу с вердиктом «всё штатно / желательна проверка / нужна ручная проверка».
  • Полное логгирование Каждый запуск пишется в Supabase: токены, стоимость, статусы этапов, HTTP-статусы, ошибки — готовая система мониторинга SLA.

⚔️ Детекция cross-user конфликтов: как это работает

Самый интересный кусок системы: LLM #3 (Critic #2) per-topic умеет отличать истинный конфликт (два разных юзера с разными результатами) от ложного (просто похожие слова в разных темах). Истинный → решение понижается до «предложено» с маркером «результаты расходятся».

• Верификация: P ✅3/⚠️5, T ✅1/⚠️4

Известные ограничения

  • Работает только с Telegram-чатами (через GramPro API)
  • Привязан к OpenRouter как LLM-провайдеру (параметризуется в Config)
  • Документация жёстко завязана на XKeen (README + FAQ с GitHub)
  • Список модераторов задан статически в Config (3 id)
  • Не поддерживает потоковые чаты (только дискретные сообщения)
  • Системные промпты (Tier-правила отбора тем, anchor-паттерны для детекта жалоб) настроены под специфику конкретного чата и его тематику — перенос на другой домен требует ручной адаптации промптов, а не только смены chat_id

📖 Глоссарий терминов

Термины, используемые в системе, в коде workflow и в выводе дайджеста.

Термин Что значит
MSG ID сообщения в Telegram. Используется для ссылок и валидации тем
anchor Самое раннее сообщение темы (от пользователя с жалобой)
anchor candidates Пред-фильтр: ~30 regex-паттернов на RU/EN, выделяют жалобы до передачи в LLM
idIndex Словарь id → {sender, isMod, isBot, preview}, строится в ноде «Форматировать для AI»
Tier 1 / Tier 2 Приоритеты включения тем. Tier 1 — всегда, Tier 2 — при наличии обсуждения
_conflict Флаг кросс-юзер конфликта: один — «помогло», другой — «не помогло»
solution Конкретное решение (команда/параметр/путь), подтверждённое автором темы
officialNote Решение из README/FAQ, отличающееся от solution
verified: true Решение явно подтверждается документацией XKeen
verified: false Решения нет в документации или оно неточное
highlight «Главное за день» — тема с максимальным msgCount
lenient parser 5-уровневый парсер JSON: снимает markdown-обёртки, чинит запятые, конвертирует Python-стиль (None/True/False → JSON)
fallback Использование данных предыдущего этапа при сбое текущего
id-fallback Принудительное восстановление оригинального id темы (LLM не имеет права менять)
digest_runs Таблица в Supabase/PostgreSQL — лог каждого прогона для аудита и SLA
[MOD] Метка в индексе сообщений, если отправитель — модератор из Config
#digest Хештег, добавляемый в публичную версию (без диагностики)
