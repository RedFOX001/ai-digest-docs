🚀 Обзор

Обзор

Это n8n-воркфлоу, который каждую ночь забирает историю Telegram-чата поддержки за прошедшие МСК-сутки, прогоняет через 5 LLM-вызовов с двумя независимыми критиками и верификацией по официальной документации, и публикует готовый структурированный дайджест — в чат модераторов и в публичный канал.

Ключевое отличие от простых AI-суммаризаторов — проверяемость: каждая тема обязана иметь ссылку на реальное сообщение, каждое решение проверяется отдельным LLM-критиком, а затем сверяется с документацией проекта.

Как это работает

  1. Сбор и фильтрация данных
    Workflow забирает историю чата через Telegram GramPro API, отфильтровывая сообщения ботов, предыдущие дайджесты (по маркеру 📋 Дайджест и 🔗) и сообщения вне временного окна. После фильтрации формируется структурированный лог с индексом сообщений и списком anchor-кандидатов — сообщений с признаками жалоб.
  2. Извлечение тем (LLM #1 — DeepSeek V4 Flash)
    Модель получает индекс, anchor-кандидаты и полный лог чата. Извлекает до 12 проблем, 5 советов и релизы по строгим Tier-правилам. Поле solution на этом этапе запрещено — проставляется только при явном подтверждении от автора темы.
  3. Критик #1 (LLM #2)
    Второй LLM-вызов проверяет JSON от экстрактора: удаляет слабые темы, сливает дубли по первопричине, валидирует id по индексу сообщений, проверяет дубликаты. Если критик вернул менее 50% тем — срабатывает fallback на данные до критика.
  4. Критик #2 per-topic (LLM #3)
    Отдельный LLM-запрос на каждую тему с контекстным окном (20 сообщений до anchor, 80 после + все сообщения автора темы). Ишет кросс-юзер конфликты, проверяет соответствие решения логу, валидирует заголовок. Принудительно восстанавливает оригинальный id, если LLM его изменил.
  5. Верификация по документации (LLM #4)
    Четвёртый LLM-вызов сверяет каждое решение/совет с официальной документацией проекта (README + FAQ с GitHub). verified=true — только если в документации есть точное совпадение. При конфликте (маркер «расходятся») — принудительно verified=false.
  6. Финальный генератор (LLM #5)
    Пятый LLM-вызов собирает HTML-текст для Telegram по жёсткому шаблону с разделами «Главное за день», «Новости и релизы», «Проблемы и решения», «Советы дня». Каждый пункт содержит ссылку на исходное сообщение.
  7. Код-валидатор (0 токенов, без LLM)
    Детерминированная проверка финального текста: замена Markdown на HTML, проверка баланса тегов, удаление пустых разделов, проверка на заглушки, схлопывание лишних пустых строк.
  8. Логгирование и отправка
    В таблицу digest_runs (Supabase/PostgreSQL) записывается полная информация о запуске: статусы этапов, токены, стоимость, ошибки. Затем дайджест отправляется в чат модераторов и, после очистки диагностики, в публичный канал.
🎯 Правила и фильтрация

🎯 Правила отбора тем: Tier 1 и Tier 2

Система фильтрует сообщения по двухуровневой системе приоритетов до того, как LLM начнёт извлекать темы. Это не решение модели — это жёсткие правила в промпте экстрактора, которые отсекают шум и выделяют то, что действительно стоит внимания модератора.

Tier 1 — ВСЕГДА включать

Даже при одном участнике и без ответов, если есть любой из маркеров:

МаркерПримерНЕ пример
Модель устройства + тех. симптомKN-2310 + SIGSEGV«у меня KN-1810, всё ок»
Код ошибки + контекстcurl response code: 400 в логе xkeen«ошибка какая-то»
Версия + явная проблемаxray 26.6.1 + вылетает«версия последняя, нормально»

Допустимые симптомы в Tier 1: ошибка, краш, не стартует, SIGSEGV, 504, таймаут, зависает.
НЕ считаются симптомами: «не работает», «проблема», «не получается».

Tier 2 — Включать при наличии обсуждения

Достаточно одного из критериев:

3+ участника

Минимум 3 автора в теме

2+ с конфигом/скриншотом

Детальное техническое обсуждение

1+ с конкретным решением

Модератор или автор предложили решение

Инфраструктурная ситуация

РКН, VPS недоступен, бан IP — даже при 2 участниках

⚠️ Важно: подтверждение от автора может прийти значительно позже в логе (10–100 сообщений спустя). Тема не считается «без подтверждения», если ответ автора просто далеко от жалобы.

Что НЕ включается в дайджест

  • ❌ 1 участник + нет ответов + нет маркеров Tier 1
  • ❌ Общие вопросы без симптома («как настроить X?», «где почитать?»)
  • ❌ Упоминание модели/версии без проблемы («у меня KN-1810, всё ок»)
  • ❌ Tip с одной голой ссылкой без описания ЧТО делает инструмент
  • ❌ Решение/воркэраунд от третьего лица без подтверждения автора темы

⚙️ Архитектура

Архитектурные особенности

  • Факт-чекинг через idКаждая тема обязана иметь ссылку на реальное сообщение в чате. LLM не может добавить факт без подтверждения — id проверяется по индексу сообщений на нескольких этапах.
  • Детекция кросс-юзер конфликтовЕсли один пользователь пишет «помогло», а другой в той же теме «не работает» — Critic #2 per-topic понижает статус до «предложено» с маркером «результаты расходятся».
  • Верификация по официальной документацииЧетвёртый LLM-вызов сверяет каждое решение с README и FAQ проекта. Подтверждённые и неподтверждённые решения явно разделены (verified true/false).
  • Принудительная защита idНормализатор ответов Critic #2 принудительно восстанавливает оригинальный id, если LLM его изменил — несмотря на прямой запрет в промпте.
  • Fallback-системаПри сбое любого LLM-этапа используются данные предыдущего этапа. Критик #1 работает с continueOnFail. Дайджест не теряется при единичных сбоях.
  • Прозрачная диагностикаВ футере каждого дайджеста — статус по каждому этапу с вердиктом «всё штатно / желательна проверка / нужна ручная проверка».
  • Полное логгированиеКаждый запуск пишется в Supabase: токены, стоимость, статусы этапов, HTTP-статусы, ошибки — готовая система мониторинга SLA.

⚔️ Детекция cross-user конфликтов: как это работает

Самый интересный кусок системы: LLM #3 (Critic #2) per-topic умеет отличать истинный конфликт (два разных юзера с разными результатами) от ложного (просто похожие слова в разных темах). Истинный → решение понижается до «предложено» с маркером «результаты расходятся».

  • Время чтения — без дайджеста ~10 мин/день на скролл 145+ сообщений, с дайджестом ~1–2 мин на 12 тем по одной строке
  • Контекст — без дайджеста нужно скроллить вверх-вниз, чтобы понять, что решилось, а что висит. Дайджест даёт это в одном экране.

Что НЕ пропустишь в дайджесте

Дайджест гарантированно подсвечивает то, что в чате часто теряется:

  • 🏆 Releases — новые версии XKeen/Xray (иначе можно месяц не замечать анонс)
  • ⚠️ Cross-user конфликты — когда один юзер пишет «помогло», а другой «не работает» (в чате это выглядит как 2 обычных сообщения в разных ветках)
  • 🔗 Решения из официальной документации — LLM #4 находит в README/FAQ то, что мимо чата вообще не проходило
  • 📌 Советы дня — пользовательские находки (порты, генераторы, патчи), которые в чате разбросаны по разным дням
  • Топ-1 проблема дня (highlight) — выбирается по msgCount, не по времени публикации (без дайджеста новая проблема тонет в потоке)

📐 Структура дайджеста: 4 секции, точные правила

Дайджест всегда собирается в одном и том же порядке. Секции могут отсутствовать, если соответствующий массив пустой — это не ошибка, а явное правило рендеринга.

Порядок и условия появления секций

СекцияЭмодзиУсловие появленияИсточник данных
Самое обсуждаемое за день🏆problems.length > 0highlight — вытаскивается из problems[] после сортировки по msgCount
Новости и релизы📣releases.length > 0Массив releases[] целиком
Проблемы и решения⚠️problems.length > 0 (остаток после highlight)problems[] после удаления highlight, отсортированы по msgCount (убыв.)
Советы дня📌tips.length > 0Массив tips[] целиком (до 5 шт.)

Как формируется highlight

highlight — это самая обсуждаемая проблема дня: берётся первый элемент из problems[] после сортировки по msgCount (по убыванию) и удаляется из основного массива через shift(). Это аппаратная логика в ноде «Подготовить финальный запрос» — финальный LLM не выбирает highlight сам. 

verifiedTopics.problems.sort((a, b) => Number(b.msgCount) - Number(a.msgCount));
highlight = verifiedTopics.problems.shift();
verifiedTopics.highlight = highlight;

Каждый пункт в дайджесте получает один из 3 значков в зависимости от комбинации трёх полей. Полная спецификация:

Формат одной записи

Каждый пункт в секциях рендерится по жёсткому шаблону:

• <b>{title}</b> [id:{messageId}] — {summary}
  {Статус}: {text решения} [{значок}]

Гарантии: у каждой записи обязательно есть [id:{messageId}] — это ссылка на реальное сообщение в чате. Если id нет — тема пропускается целиком, а не выводится с пустой ссылкой. Именно так обеспечивается «без выдумок».

Разбиение длинного дайджеста на части

Telegram ограничивает длину одного сообщения (≈4096 символов), поэтому длинные дайджесты автоматически делятся на части:

  • MAX_LEN = 3500 символов на часть (с overhead на header/footer ~100 символов)
  • Граница разбиения — между темами (по маркеру • <b>), тема никогда не разрезается пополам
  • Header (только в первой части): 📋 Дайджест за {date} + 📨 Сообщений обработано: {N}
  • Continuation header (в остальных): 📋 Дайджест за {date} (продолжение)
  • Footer (только в последней части): модель + токены + стоимость + блок диагностики ⚙️
  • Метка части: Часть N из M добавляется только если M > 1

ORDER BY cost DESC NULLS LAST
LIMIT 5;

Все случаи fallback'а (по любому этапу):

SELECT digest_date, final_status, failed_stages
FROM digest_runs
WHERE critic_fallback_used = TRUE
   OR verif_fallback_used  = TRUE
   OR 'verifier' = ANY(failed_stages)
ORDER BY run_at DESC;

Распределение по статусам за месяц:

SELECT final_status, COUNT(*) AS runs
FROM digest_runs
WHERE run_at > NOW() - INTERVAL '30 days'
GROUP BY final_status
ORDER BY runs DESC;

Известные ограничения

  • Работает только с Telegram-чатами (через GramPro API)
  • Привязан к OpenRouter как LLM-провайдеру (параметризуется в Config)
  • Документация жёстко завязана на XKeen (README + FAQ с GitHub)
  • Список модераторов задан статически в Config (3 id)
  • Не поддерживает потоковые чаты (только дискретные сообщения)
  • Системные промпты (Tier-правила отбора тем, anchor-паттерны для детекта жалоб) настроены под специфику конкретного чата и его тематику — перенос на другой домен требует ручной адаптации промптов, а не только смены chat_id

На данный момент сбои единичные, идёт накопление статистики — пока инцидентов недостаточно, чтобы оправдать затраты на реализацию детектора CJK-диапазона, ручных retry-циклов (n8n не поддерживает semantic retry), новых колонок в digest_runs и изменения finalStatus. 🟡 Средний приоритет

ИдеяПочему не реализовано
t31 — Сброс resolved для tips Пока не актуально — на практике LLM крайне редко проставляет resolved для советов дня. Как только появятся кейсы с ложными ✅ в tips[] — задача будет поднята.
t15 — Косвенные подтверждения resolved Сложная семантическая задача: как алгоритмически отличить «замолчал, потому что решилось» от «замолчал, потому что ушёл». Требует анализа паттернов умолчания и пороговых значений, что пока не проработано.
t17 — Coverage-проверка Отложен до накопления данных — требуется история прогонов в digest_runs (t26), чтобы построить baseline «нормального» количества тем для данного объёма сообщений. Без статистики за 2–4 недели алерт будет либо молчать, либо спамить.
t19 — Переклассификация tips→problems Пока не актуально — на практике советы дня редко перерастают в полноценные проблемы с обсуждением. Если появятся кейсы, где tip набрал 3+ участников с жалобой — задача будет пересмотрена.
t23 — Highlight в промпт экстрактора Критерии описаны, но сама механика не внедрена. Текущая аппаратная сортировка по msgCount работает предсказуемо и надёжно. Перенос логики в LLM добавит неопределённости, расход токенов и потребует дополнительной верификации выбора.
t25 — Разрешить Критику #1 добавлять темы Пока не встречалось примеров необходимости — экстрактор стабильно не пропускает критически важные темы. Если появится кейс, где Critic #1 в переданном контексте видит явную жалобу, которую Extractor упустил — задача будет актуализирована.
t33 — Rescue Extractor Требует отдельной ноды + дублирующего HTTP-вызова к LLM с усиленным промптом. Пока не было инцидентов, которые бы это оправдали (риск экстенсивного расхода токенов при редких сбоях не перевешивает сложность реализации).
t42 — Не удалять highlight из problems[] механически Отложен, потому что текущий shift() после сортировки по msgCount предсказуем и надёжен. Перенос выбора highlight в LLM — качественное изменение: модель может ошибиться в выборе «главного», и тогда придётся добавлять верификацию. Пока решили не усложнять.
t46 — Синхронизировать контекстное окно Critic #2 Чисто косметическая синхронизация промпта (там написано «60 сообщений после») с кодом (передаётся 80). Статус 🟢 LOW — не влияет на работу, просто рассогласование документации.
